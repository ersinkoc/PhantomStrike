# Web Application Fingerprinting

## Overview
Web application fingerprinting identifies the technologies, frameworks, content management systems, web servers, and security mechanisms powering a target website. Accurate fingerprinting directs subsequent testing toward known vulnerability classes, default credentials, and technology-specific exploit techniques.

## Classification
- **MITRE ATT&CK:** T1592.004 (Client Configurations), T1595.002 (Vulnerability Scanning)
- **Phase:** Reconnaissance
- **Risk Level:** Low to Medium (sends HTTP requests to target)
- **Prerequisites:** Target URL(s), HTTP/HTTPS access

## Detection Methodology

### 1. HTTP Response Header Analysis
Extract technology indicators from server responses:
- `Server`: Web server software and version (Apache, Nginx, IIS, LiteSpeed)
- `X-Powered-By`: Backend language/framework (PHP, ASP.NET, Express)
- `X-AspNet-Version` / `X-AspNetMvc-Version`: ASP.NET version disclosure
- `X-Generator`: CMS identification (WordPress, Drupal, Joomla)
- `X-Drupal-Cache` / `X-Drupal-Dynamic-Cache`: Drupal indicators
- `Set-Cookie`: Session cookie names reveal frameworks (PHPSESSID, JSESSIONID, ASP.NET_SessionId, connect.sid, _rails_session)
- `X-Request-Id` / `X-Correlation-Id`: Backend architecture hints
- Missing security headers: absence of CSP, HSTS, X-Frame-Options

### 2. HTML and DOM Analysis
Inspect page source for technology signatures:
- Meta tags: `<meta name="generator" content="WordPress 6.x">`
- CSS/JS file paths: `/wp-content/`, `/sites/default/files/`, `/static/js/`
- HTML comments: `<!-- generated by -->`, build timestamps
- Script tags: framework-specific globals (`React`, `Angular`, `Vue`, `jQuery`)
- Form action URLs and hidden fields reveal backend technology
- Error pages: default error pages expose server/framework

### 3. WAF Detection
Identify Web Application Firewalls protecting the target:
- Custom error pages on malicious input (403, 406, 429)
- Modified response headers (`X-Sucuri-ID`, `cf-ray`, `X-CDN`)
- Cookie injection (WAF session cookies)
- Behavioral analysis: send common attack payloads and observe blocking
- Known WAF signatures: Cloudflare, AWS WAF, Akamai, Imperva, Sucuri, ModSecurity, F5

### 4. CMS Identification
Detect and version content management systems:
- **WordPress:** `/wp-login.php`, `/wp-admin/`, `/wp-content/`, `/wp-includes/`, `meta generator`
- **Drupal:** `/core/misc/drupal.js`, `CHANGELOG.txt`, `/sites/default/`, `Drupal.settings`
- **Joomla:** `/administrator/`, `/components/`, `/modules/`, `meta generator`
- **Magento:** `/skin/frontend/`, `/js/mage/`, admin panel at `/admin`
- **SharePoint:** `/_layouts/`, `MicrosoftSharePointTeamServices` header
- **Custom CMS:** analyze URL patterns, admin panels, file structures

### 5. JavaScript Framework Detection
Identify client-side frameworks and libraries:
- React: `data-reactroot`, `__NEXT_DATA__`, `_next/static`
- Angular: `ng-version`, `ng-app`, `.angular` paths
- Vue.js: `__vue__`, `data-v-` attributes, `/js/chunk-` paths
- jQuery: `jquery.min.js` includes, `$.fn.jquery` version
- Bootstrap: bootstrap CSS/JS includes, class naming patterns
- Check `window.__REACT_DEVTOOLS_GLOBAL_HOOK__` and similar globals

### 6. Server Technology Fingerprinting
Determine server-side technology through behavioral analysis:
- Default file extensions: `.php`, `.asp`, `.aspx`, `.jsp`, `.py`, `.rb`
- URL patterns: query strings vs. path-based routing
- Error handling: stack traces, debug pages, 404 styles
- Case sensitivity: Linux (case-sensitive) vs. Windows (case-insensitive)
- File enumeration: probe for `web.config`, `.htaccess`, `robots.txt`, `sitemap.xml`

## Tool Usage

### whatweb
```bash
# Basic fingerprint
whatweb target.com

# Aggressive mode (more requests, deeper detection)
whatweb -a 3 target.com

# Verbose output with all plugins
whatweb -v target.com

# JSON output for parsing
whatweb --log-json=output.json target.com

# Scan multiple targets
whatweb -i urls.txt --log-json=output.json

# Specific plugins only
whatweb --plugins "HTTPServer,PoweredBy,MetaGenerator" target.com

# With custom User-Agent
whatweb -U "Mozilla/5.0" target.com

# Follow redirects
whatweb --follow-redirect=always target.com
```

### Wappalyzer CLI (wappalyzer-cli / webtech)
```bash
# Detect technologies via webtech (Wappalyzer-compatible)
webtech -u https://target.com

# JSON output
webtech -u https://target.com --json

# Multiple URLs from file
webtech -ul urls.txt --json

# Using wappalyzer npm package
npx wappalyzer https://target.com

# Verbose detection
npx wappalyzer https://target.com --pretty
```

### wafw00f
```bash
# Detect WAF on target
wafw00f https://target.com

# Test all WAF signatures
wafw00f https://target.com -a

# Verbose mode
wafw00f https://target.com -v

# Multiple targets from file
wafw00f -i urls.txt

# JSON output
wafw00f https://target.com -o output.json -f json

# List all detectable WAFs
wafw00f -l
```

### httpx
```bash
# Probe hosts for HTTP services with technology detection
httpx -l hosts.txt -tech-detect -status-code -title -o results.txt

# Full fingerprinting suite
httpx -l hosts.txt -tech-detect -status-code -title -server -content-length \
  -follow-redirects -json -o results.json

# With response header capture
httpx -l hosts.txt -include-response-header -json -o full_results.json

# Screenshot and hash
httpx -l hosts.txt -screenshot -favicon -hash md5

# Filter by status code
httpx -l hosts.txt -mc 200,301,302,403 -tech-detect -title

# Web server header extraction
httpx -l hosts.txt -server -content-type -json

# Custom ports
httpx -l hosts.txt -ports 80,443,8080,8443,8000,3000 -tech-detect
```

### Manual header inspection
```bash
# Fetch headers with curl
curl -sI https://target.com

# Fetch headers following redirects
curl -sIL https://target.com

# Check specific technology indicators
curl -s https://target.com | grep -i "generator\|powered\|framework\|version"

# Test for default files
curl -s -o /dev/null -w "%{http_code}" https://target.com/robots.txt
curl -s -o /dev/null -w "%{http_code}" https://target.com/wp-login.php
curl -s -o /dev/null -w "%{http_code}" https://target.com/sitemap.xml
```

## Output Analysis Tips
- **Version precision matters:** Generic detection (e.g., "Apache") is less useful than versioned detection (e.g., "Apache 2.4.49"). Versioned results map directly to CVE lookups.
- **WAF presence changes strategy:** When a WAF is detected, switch to WAF bypass techniques before vulnerability scanning. Test bypass with benign payloads first.
- **CMS version to exploit chain:** WordPress, Drupal, and Joomla versions map directly to known exploits. Check plugin/theme versions as well since they are the most common attack vector.
- **Multiple technology layers:** Modern applications stack multiple technologies (e.g., Nginx reverse proxy, Node.js backend, React frontend, PostgreSQL database). Map the full stack.
- **False negatives:** Some sites aggressively strip headers and fingerprints. Use multiple tools and techniques for confirmation.
- **Cookie analysis:** Session cookie names are highly reliable indicators: `PHPSESSID` = PHP, `JSESSIONID` = Java, `ASP.NET_SessionId` = .NET, `connect.sid` = Node.js/Express.
- **CDN detection:** Cloudflare, Akamai, and Fastly mask the origin server. Look for origin IP leaks through DNS history, email headers, or direct IP access.

## Evidence Collection
- Complete technology stack summary (server, language, framework, CMS, WAF, CDN)
- HTTP response headers from key endpoints
- Detected software versions with CVE cross-references
- WAF identification results with bypass feasibility assessment
- CMS detection with plugin/theme enumeration where applicable
- Screenshots of technology-revealing pages (error pages, admin panels)
- JSON output from all fingerprinting tools for automated processing
